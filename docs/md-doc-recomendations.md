# Рекомендации по документации UI Kit (Markdown + сайт + LLM)

## 1. Короткий вывод

Для вашего проекта оптимален **гибридный подход**:

1. Оставить текущие интерактивные демо-блоки (`app-usage-demo`) для разработчиков.
2. Добавить **канонические docs-страницы** формата “1 компонент = 1 страница”.
3. API/типы и часть примеров генерировать автоматически из кода.
4. Делать docs-страницы максимально доступными для индексации и LLM (SSR/SSG + markdown-слой + индекс для ИИ).

Это лучше, чем:

- только текущий режим табов (удобно, но хуже для SEO/LLM и долгосрочной структуры),
- или только “чистый markdown без живых примеров” (лучше читаемость, но хуже DX и доверие к примерам).

## 2. Сравнение вариантов

### Вариант 1: как сейчас (демо + табы с markdown-сниппетами)

Плюсы:

- Быстрый DX, удобно копировать куски кода.
- Минимум изменений.

Минусы:

- Контент в табах и в runtime-рендере слабее индексируется.
- Нет единой “канонической” страницы компонента.
- Легче получить рассинхрон между “примером на экране” и “описанием компонента”.

### Вариант 2: полностью markdown (1 файл = 1 страница)

Плюсы:

- Хорошая читаемость, хорошая индексация.
- Ясная структура знаний для команды и LLM.

Минусы:

- Теряется интерактивность или становится сложнее поддерживать.
- Примеры чаще устаревают, если не автоматизировать.

### Вариант 3 (рекомендуется): гибрид

Плюсы:

- И интерактив, и качественный reference-слой.
- Лучший баланс для разработчиков, поиска и LLM.
- Легче масштабировать на десятки компонентов.

Минусы:

- Нужна дисциплина и легкая автоматизация (линки, автогенерация, чек-листы).

## 3. Целевая архитектура документации

## 3.1. Канонический источник по компоненту

На каждый компонент должна быть одна основная docs-страница:

- `docs/components/<component>/index.md`

Дополнительно:

- `docs/components/<component>/examples/*.md` (если нужен разнос примеров),
- `docs/components/<component>/a11y.md` (если много требований доступности),
- `docs/components/<component>/migration.md` (при breaking changes).

## 3.2. Что остается на сайте

Оставить текущий интерактивный слой (`UsageDemo`) для playground/preview:

- живой компонент,
- вкладки кода (`html`, `ts`),
- быстрый copy-paste.

Но вокруг него должна быть полноценная docs-страница с текстом, API и best practices.

## 3.3. Что генерировать автоматически

Автоматизировать:

- API reference (inputs/outputs/events/types),
- таблицы props,
- приоритетные сниппеты (где возможно из source).

Ручной markdown должен описывать:

- когда использовать,
- поведение/ограничения,
- accessibility и UX-правила,
- do/don’t,
- edge cases.

## 4. Стандарт страницы компонента (шаблон)

Рекомендуемая структура `index.md`:

1. `# ComponentName`
2. Краткое назначение (2-4 предложения).
3. `## Когда использовать`
4. `## Быстрый старт`
5. `## Примеры`
6. `## API` (ссылка/вставка автогенерации)
7. `## Accessibility`
8. `## Стилизация и темы`
9. `## Ограничения и edge cases`
10. `## FAQ`
11. `## История изменений` (опционально)

## 5. Стандарт markdown-файлов

## 5.1. Frontmatter (рекомендуется)

```yaml
---
title: Select
description: Single-value select with keyboard navigation and form integration.
component: select
status: stable
since: 1.4.0
tags: [forms, selection, accessibility]
---
```

## 5.2. Правила контента

- Один `h1` на страницу.
- Логичные `h2/h3`, без скачков уровней.
- У каждого code block указывать язык (`ts`, `html`, `css`, `bash`).
- Описывать не только “как”, но и “когда/почему”.
- Указывать ограничения и known pitfalls.
- Для примеров добавлять ожидаемый результат и поведение с клавиатурой.

## 5.3. Правила ссылок

- Стабильные URL и anchor.
- Не использовать “битые относительные” ссылки.
- Проверять ссылки в CI.

## 6. Требования для хорошего чтения LLM

## 6.1. Что важно технически

- Критичный контент должен быть доступен в серверном HTML (SSR/SSG/prerender).
- Не прятать важные разделы только за интерактивом.
- Семантическая структура HTML (`article`, `nav`, `h1..h3`, списки, таблицы).
- Стабильные slug/anchors.

## 6.2. Отдельный AI-friendly слой

Добавить:

- `llms.txt` (краткий индекс по документации),
- `llms-full.txt` (расширенный индекс),
- при возможности markdown-версию канонических страниц по предсказуемому пути.

## 6.3. Robots/индексация

- Не блокировать docs-страницы для основных ботов поиска/ИИ, если это публичная документация.
- Проверить `robots.txt` и отсутствие случайных `noindex` на docs-маршрутах.

## 7. Как применить к текущему проекту (практически)

Текущее состояние уже хорошее для старта: есть `UsageDemo` и markdown-сниппеты в `public/documentation`.

Рекомендуемый переход:

1. Сохранить текущие `codeBlocks` и рендер табов в `UsageDemo`.
2. Добавить “канонические docs-страницы” для каждого компонента (начать с `select`, `input`, `checkbox`, `switch`).
3. В каждой странице встроить существующие живые демо.
4. Создать единый раздел `API`, который пополняется автогенерацией.
5. Добавить индекс для LLM (`llms.txt`) и человекочитаемый docs index.

## 8. Пошаговый план миграции

## Этап 1 (быстрый, 1-2 дня)

- Утвердить единый шаблон `index.md`.
- Привести 2-3 компонента к шаблону.
- Ввести checklist в PR: “обновлена docs-страница + пример + API”.

## Этап 2 (системный, 3-5 дней)

- Подключить автогенерацию API docs.
- Добавить CI-проверки:
  - битые ссылки,
  - пустые/сиротские markdown-файлы,
  - несоответствие списка компонентов и docs.

## Этап 3 (LLM/SEO, 1-2 дня)

- Добавить `llms.txt` и `llms-full.txt`.
- Проверить SSR/SSG/prerender для docs-раздела.
- Проверить индексируемость docs-страниц.

## 9. Критерии качества (Definition of Done)

Документация компонента считается готовой, если:

1. Есть каноническая страница с назначением и ограничениями.
2. Есть минимум один live example и минимум один copy-paste пример.
3. API-слой актуален и не руками синхронизируется.
4. Описаны accessibility-поведение и keyboard navigation.
5. Страница доступна для индексации и читается без JS-магии.

## 10. Что выбрать сейчас

Решение: **внедрять вариант 3 (гибрид)**.

Почему:

- минимальный риск для текущей разработки,
- не ломает существующие demo flow,
- дает лучший результат по качеству документации, SEO и пригодности для LLM.
